## 1 Background
Australia boasts some of the most unique and diverse wildlife on the planet. A machine
learning-based, vision-driven platform plays a critical role in citizen science, allowing
researchers, students, and nature enthusiasts to capture, log, and identify fauna across
the continent. Think about all the fantastic stuff captured: stunning shots of wildlife in
action, unique recordings of calls, and insightful video clips of behaviour. This is a gold
mine for environmental research, helping to identify species, track individuals, study
behaviour, monitor populations, and ultimately protect our ecosystems.

However, imagine trying to find a specific photo of a rare species from last year if it's buried on
someone's random hard drive, or losing a whole batch of important video recordings because a
memory card got corrupted. Data scattered everywhere is data that's hard to use, easy to lose,
and impossible to share properly. A good storage system means everything is safe, backed up,
and organised in one central location, ensuring that any member who needs data can find it
efficiently. Collaborating on projects becomes a breeze because everyone looks at the same
organised data, ensuring valuable information isn't lost.

This assignment aims to build a multi-cloud online storage system that allows users to store,
organise, and retrieve media files based on auto-generated and manual tags. The focus of this
project is to design a serverless application across at least two major cloud providers (e.g.,
AWS, GCP/Azure/Oracle) that enables clients to upload their media files (images and videos)
to public cloud storage. Upon upload, the application automatically tags the file with the
species detected in it using pre-trained ML models, automatically sorts/moves the file to the
corresponding directory, and adds a record in a database for searching.

## 2.1 Assignment Description
Teams should develop a multi-cloud solution that leverages serverless compute, RESTful API,
storage, and database services across multiple providers (e.g., AWS, GCP, Oracle or Azure) to
build a system for automated wildlife tagging and query processing. The teams should produce
a solution that enables end-users to upload their files to a cloud storage bucket. Upon
uploading a file to the designated bucket, a serverless function is automatically triggered, using
the provided model to identify the species in the file and store the list of detected tags, along
with the file's storage URL, in a highly available database. Furthermore, the end user should be
able to submit queries to a RESTful API endpoint to search for tagged files.

3 Authentication and Authorisation (10%)
Security is one of the most crucial aspects of developing cloud applications. When your
application is publicly exposed, you must ensure that your endpoints and resources are
safeguarded against unauthorised access and malicious requests. AWS, through its Cognito
service, provides a straightforward, secure, and centralised approach to protect your web

Note on Multi-Cloud: For user authentication and authorisation, you must use AWS
Cognito. Other services can be hosted in other cloud services.  You may need to consider how
to securely pass AWS Cognito-generated authentication tokens to authorise requests to your
secondary cloud provider's API endpoints or serverless functions.
Your application should support the following workflow and features:
## ●
Detect whether a user is authenticated. If the user has not signed in, access to all pages

## ●

## 4 Core Functionalities (50%)
## 4.1 Model Handling
Machine learning and deep learning models are often updated to improve their quality. If a new
version of the pretrained model becomes available, your solution design should be flexible
enough to use it without changing your source code/lambda function.

## 4.2 File Handling
Your solution should provide a mechanism to upload a file to cloud storage (e.g., an S3 bucket).
Uploading a file can be done either through a RESTful API endpoint or directly using cloud
SDKs. Whenever a file is uploaded, your system must trigger an event and invoke a serverless
function.
## ●
Deduplication: Users may upload the same file repeatedly, wasting storage space. Your
application should have a mechanism to prevent duplicate uploads using a file checksum.
## ●
Building Image Thumbnails: Upon uploading an image, a serverless function should
generate a thumbnail for it. Creating a thumbnail involves resizing the original image to a
smaller dimension while maintaining its aspect ratio, then compressing it to reduce file
size. You can use the OpenCV library for this purpose. You are expected to configure the
triggers and grant the required resource permissions (execution roles) for the function.

## ●
Tagging Files: Another serverless function should be triggered to read the uploaded file,
detect species using the ML model, and save the file type and a list of detected tags,
along with the storage URLs for the file and its thumbnail (for images), in a database for
future queries.
● Video files can be handled by extracting frames as images, then using the image
detection model. For consistency and reduction of data processing, please extract 1

image per second (do not try to extract all frames).
## 4.3 Queries
Your application should provide APIs that allow the following queries:
## ●
Find images and videos based on the tags: The user can send a JSON-based query to
request URLs of files that contain specific tags with minimum counts (e.g., {"koala": 3},
{"wombat": 2, "magpie": 1}, which shows all files with >=3 koalas in the first case and >=2
wombats AND >=1 magpie in the second example). You are expected to create a RESTful
API allowing users to submit requests. A response should include the list of URLs to all
thumbnails of images and full URLs for videos that contain all the requested tags. Your
query may require triggering a function that receives a list of tags from the database, i.e.,
logical AND operation, not OR operation between tags. In your UI, you can preview the
thumbnails found as the results (to save bandwidth), and users can request the full-size
image by clicking on them.


## ●
Find files based on species: User can send a request similar to the previous query
without count, e.g. {"dingo"}. Your application will return all images and videos containing
at least one dingo.


## ●
Find files based on the thumbnail's URL: Users can input the URL of a thumbnail into the
system and receive the corresponding full-size image URL.


## ●
Find files based on the tags of a file: The user can send a file as part of an API call. The
list of all species (tags) in the sent file is discovered, and then all the files in the database
containing that set of tags are found. As a response, the list of URLs of the matching
media files is returned to the user. Ensure that the file uploaded for query purposes is not
added to the database or permanently stored.


## ●
Manual addition or removal of tags with bulk tagging: Your solution should provide a
POST API that enables end-users to manually add or remove tags from files. The JSON
message sent must include a list of URLs, the tags to modify, and an "operation" key set to
1 (for adding a tag) or 0 (for removing a tag). If a tag is not included in the list of tags
requested for deletion, simply ignore it.


## ●
Delete files: The user can send a list of URLs to an API, and the system should remove
the files and their thumbnails from storage and all relevant entries from the database.


## 4.4 Tag-based Notifications
Users can receive email notifications or updates related to files based on specific tags. For
example, users can receive notifications when new images with specific species are added or
updated to the system. You can use AWS SNS or a similar serverless messaging equivalent on

your secondary cloud for this purpose.
